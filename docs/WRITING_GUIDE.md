# 技术文章写作规范说明

本文档用于统一博客文章的结构、排版和表达方式，目标是提升可读性、专业感和长期维护效率。

## 1. 写作目标

技术文章应优先满足以下目标：

- 让读者快速理解文章解决的问题
- 让读者能顺着结构高效获取信息
- 让示例代码和结论具备可操作性
- 让文章在半年后回看仍容易维护和更新

## 2. 推荐文章结构

建议一篇标准技术文章至少包含以下部分：

1. 背景
2. 问题分析
3. 方案说明
4. 实现过程
5. 关键代码
6. 注意事项
7. 总结

如果文章较短，可以压缩为：

1. 背景
2. 方案
3. 实现
4. 总结

## 3. 标题层级规范

- 文章主标题由 frontmatter 中的 `title` 提供，正文中不要重复写一级标题
- 正文一级结构统一从 `##` 开始
- 子内容使用 `###`
- 不要随意跳级，例如从 `##` 直接跳到 `####`

推荐示例：

```md
## 背景

## 方案设计

### 为什么选择 Astro

### 为什么选择 MDX

## 实现过程

## 总结
```

## 4. 段落规范

- 每段只表达一个重点
- 单段不要过长，建议控制在 3 到 5 行
- 长说明拆成多个自然段
- 避免连续大段纯概念描述

建议：

- 定义概念后，尽快跟上示例
- 提出结论后，尽快给出原因

## 5. 代码示例规范

代码块必须标注语言类型。

推荐：

```ts
const posts = await getCollection("blog");
```

```bash
pnpm create astro@latest .
```

不推荐：

```txt
const posts = await getCollection("blog");
```

补充建议：

- 代码示例只保留关键部分
- 太长的代码应拆分讲解
- 示例代码要和正文结论对应
- 如有前置条件，要在代码前说明

## 6. 命令、路径、变量的排版规范

以下内容建议统一使用反引号包裹：

- 命令：`pnpm dev`
- 路径：`src/content/blog`
- 变量名：`siteConfig`
- 配置项：`draft`
- 包名：`@astrojs/mdx`

这样可提升技术内容的辨识度和扫读效率。

## 7. 列表使用规范

- 枚举步骤时使用有序列表
- 罗列特性、优缺点、注意点时使用无序列表
- 列表项尽量保持句式一致
- 不要把整篇文章写成堆叠列表

列表适合梳理信息，不适合替代正文解释。

## 8. 图片与图表规范

- 图片应服务于说明，而不是装饰
- 截图应清晰，避免过大或过模糊
- 图表适合解释流程、架构、依赖关系
- 图片下建议补一句说明

如果文章偏工程实践，优先使用：

- 流程图
- 架构图
- 对比表

## 9. 技术表达规范

建议遵循以下原则：

- 先说结论，再说原因
- 先说适用场景，再说实现细节
- 先给最小可行方案，再补充扩展方案
- 明确说明约束条件和适用边界

避免：

- 只介绍概念，不说明为什么
- 只贴代码，不解释上下文
- 只说“最佳实践”，不说明适用前提

## 10. 技术文章的常见结构模板

### 模板 A：方案分析类

适合：
- 技术选型
- 框架对比
- 工具推荐

结构：

1. 背景
2. 评估维度
3. 各方案对比
4. 最终选择
5. 总结

### 模板 B：实战教程类

适合：
- 从零搭建
- 集成某项功能
- 完整操作步骤

结构：

1. 背景
2. 目标效果
3. 前置条件
4. 步骤拆解
5. 完整代码
6. 常见问题
7. 总结

### 模板 C：问题排查类

适合：
- Bug 处理
- 性能优化
- 构建问题定位

结构：

1. 问题现象
2. 排查思路
3. 原因分析
4. 解决方案
5. 验证结果
6. 总结

## 11. 风格建议

推荐风格：

- 准确
- 克制
- 有判断
- 有证据

建议尽量做到：

- 结论明确
- 例子真实
- 不夸张
- 不堆砌术语

## 12. 发布前自检清单

发布前建议检查：

- 标题是否准确
- 摘要是否完整
- 标题层级是否合理
- 代码块是否标注语言
- 命令和路径是否用反引号包裹
- 是否有清晰结论
- 是否说明适用场景与限制
- 是否存在错别字和重复表达
